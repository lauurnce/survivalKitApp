# Verification: `classes` + `class_members` schema

This repo has no pgTAP harness; migrations are verified by hand-running SQL
against a local or live Supabase instance. This checklist defines what
"passing" means for `20260716000000_classes.sql`, per the TDD-for-SQL
approach: write this checklist first (red ‚Äî tables don't exist yet), then
the migration, then re-run this checklist (green).

## Preconditions

- `subjects` table exists with `id uuid primary key` (see `001_initial_schema.sql`).
- `years` table exists with `id uuid primary key` (see `001_initial_schema.sql`).
- `subscription_status` enum exists as `('active', 'paused', 'cancelled')`
  (see `20260623100000_subscriptions.sql`).

## Step 6 ‚Äî actually executed (no Docker/`supabase db reset` available)

This environment has no Docker, so `supabase db reset` could not run
directly. Instead, a genuine local Postgres 16 instance was started by hand
(`initdb` + `pg_ctl`, both present via Homebrew) and every migration file
under `supabase/migrations/` was replayed against it in filename order with
`psql -v ON_ERROR_STOP=1`, after stubbing the Supabase-only primitives this
repo's migrations assume (`anon`/`authenticated`/`service_role` roles,
`auth` schema, `auth.users` table, `auth.uid()`) ‚Äî none of which this task's
migration touches.

Confirmed by live execution, not just reading:

- [x] All Steps 1‚Äì5 above passed exactly against the running instance
      (table shapes, index names, FK cascade behavior, RLS enabled +
      policies present).
- [x] Case-insensitive unique `code`: inserting `'ABC123'` then `'abc123'`
      raised `duplicate key value violates unique constraint
      "classes_code_idx"` (`Key (upper(code))=(ABC123) already exists`).
- [x] `unique(class_id, device_id)` on `class_members`: duplicate insert
      raised `duplicate key value violates unique constraint
      "class_members_class_id_device_id_key"`.
- [x] `on delete cascade`: deleting the parent `classes` row removed its
      `class_members` rows (count went from 1 to 0).
- [x] RLS policies enforce per-device visibility under a non-superuser role
      with `set app.device_id = '...'`: the rep device and the joined member
      device each see the class row; an unrelated device sees zero rows from
      both `classes` and `class_members`.
- [x] Idempotency: re-running `20260716000000_classes.sql` a second time
      exits 0, emitting only `NOTICE: relation "..." already exists,
      skipping` for tables/indexes and clean `DROP POLICY`/`CREATE POLICY`
      pairs ‚Äî no errors.
- [x] Full-history replay: every migration up to and including
